# Documentation Sub-Harness

Use this file when writing or refactoring README content, technical guides, usage notes, or operator documentation.

## Skills To Activate

- `documentation-writer`
- `solid`
- `tdd`

Use `deployment-pipeline-design` alongside this file for deployment or operational docs.

## Working Rules

- Write for a clear reader and goal.
- Keep documentation actionable and accurate.
- Prefer concise, task-oriented Markdown.
- Match repo terminology and actual implementation.
- Avoid aspirational docs that describe features the code does not have.

## Documentation Types

Choose the right mode before writing:

- Tutorial: teach a newcomer through a successful outcome
- How-to: solve a specific problem
- Reference: describe exact system behavior
- Explanation: clarify why something works this way

## Avoid

- Mixing explanation and step-by-step instructions carelessly
- Documenting unimplemented behavior
- Overspecifying future architecture as if it already exists
